# 架构设计详解

本文档对应当前仓库中的 `pyrepo-test-agent` 实现。项目目标是构建一个
**真实 LLM 驱动的 Python 仓库自动测试 Agent 系统**：输入一个 Python
仓库路径，系统自动完成仓库扫描、规格推断、测试规划、并行测试生成、
pytest 执行、失败归因、有限轮次测试修复、coverage 采集和 Markdown 报告输出。

项目默认使用 ZAI GLM API，当前默认模型为 `glm-4.7`，也支持通过 CLI 切换到
`glm-5`。GLM-4.7 和 GLM-5 都按 5 个并发额度处理，系统通过共享
`asyncio.Semaphore(5)` 对所有 Agent 的模型调用进行统一限流。单次请求默认
120 秒超时，并对 429 等临时错误进行最多 3 次重试。

## 一、整体架构

```text
用户输入
  |
  v
CLI(app/main.py)
  |
  v
.env 加载 -> ZAI GLM Client 初始化 -> TestGenState 初始化
  |
  v
Repo Scanner Agent
  |
  v
Spec Inference Agent  --调用 GLM--> 行为规格
  |
  v
Test Planner Agent    --调用 GLM--> 测试计划
  |
  v
Parallel Executor
  |--------------------|----------------------|
  v                    v                      v
Unit Test Writer       API Test Writer         Regression Case Agent
  |                    |                      |
  |------调用 GLM------|------调用 GLM---------|
  |                    |                      |
  |--------------------|----------------------|
                       v
              generated_tests/
                       |
                       v
              Sandbox Runner(pytest + coverage)
                       |
                       v
              通过？
                | Yes
                v
              Report Builder
                |
                v
              report.md + coverage.json

                | No
                v
              Failure Analyst
                |
                v
              Repair Agent --调用 GLM--> 修改生成测试文件
                |
                v
              重新执行 pytest，直到通过或达到最大修复轮次
```

## 三、核心状态对象

系统使用共享状态对象 `TestGenState` 串联所有 Agent。它不是一个松散的大字典，
而是由多个 dataclass 组成的结构化状态。

关键字段包括：

- `repository_info`：仓库路径、Python 文件、测试文件、模块依赖图、目标符号、API 路由。
- `inferred_specs`：GLM 推断出的函数/类/API 行为规格。
- `test_plan`：GLM 生成的 unit/api/regression 测试计划。
- `generated_unit_tests` / `generated_api_tests` / `generated_regression_tests`：生成测试文件路径。
- `run_results`：pytest 每轮执行结果，包括命令、返回码、stdout、stderr、耗时。
- `failure_reports`：失败归因结果。
- `coverage_summary` / `coverage_report_path`：coverage 汇总和 JSON 报告路径。
- `warnings` / `error_messages` / `agent_events`：可观测性信息。

这种共享状态设计适合当前 Pipeline，因为各阶段有清晰的先后关系，Writer Agent
写入的字段也相互独立。

## 四、Agent 设计

### 4.1 BaseAgent

所有 Agent 继承 `BaseAgent`。父类固定执行模板：

1. 记录开始事件。
2. 调用子类 `execute()`。
3. 成功时记录耗时。
4. 失败时把错误写入 `state.error_messages`。

这样新增 Agent 时只需要关注业务逻辑，不需要重复写日志和异常边界。

### 4.2 Repo Scanner Agent

职责：

- 扫描 Python 文件。
- 过滤 `.git`、`.venv`、`__pycache__`、`output` 等目录。
- 识别现有测试文件。
- 使用 AST 分析函数、类、装饰器、签名、docstring、返回注解。
- 识别 FastAPI 风格路由装饰器，例如 `@app.get("/path")`。
- 构建模块 import 图。
- 归一化 changed files。

该阶段不调用 LLM，因为它是确定性的静态分析层。

### 4.3 Spec Inference Agent

职责：

- 为每个目标符号提取源码上下文。
- 调用 GLM 推断行为规格。
- 要求模型输出 JSON：

```json
{
  "behavior": "...",
  "boundaries": ["..."],
  "dependencies": ["..."],
  "risk": "low|medium|high"
}
```

输出会解析为 `InferredSpec`，供 Test Planner 和 Writer 使用。

### 4.4 Test Planner Agent

职责：

- 汇总目标符号、变更文件和规格信息。
- 调用 GLM 生成结构化测试计划。
- 支持三类测试：
  - `unit`
  - `api`
  - `regression`
- 对模型返回进行校验，只接受有效 `symbol_id` 和合法测试类型。

模型输出格式：

```json
{
  "plan": [
    {
      "symbol_id": "calculator.py::add",
      "test_type": "unit",
      "priority": 1,
      "rationale": "覆盖核心加法行为"
    }
  ]
}
```

### 4.5 并行 Writer Agents

Writer 阶段由 `ParallelExecutor` 使用 `asyncio.gather` 并行运行：

- `UnitTestWriterAgent`
- `ApiTestWriterAgent`
- `RegressionCaseAgent`

三个 Agent 共享同一个 `ZaiChatClient`，因此即使并行执行，所有模型请求也会被
同一个信号量限制在最多 5 个并发。

Writer 的共同约束：

- 只输出 Python 代码。
- 生成 pytest 文件。
- 必须包含仓库路径注入，确保目标仓库模块可导入。
- 不修改业务源码。
- 生成结果会先提取 Python 代码块，再做语法和 pytest 形态校验。

### 4.6 Sandbox Runner Agent

职责：

- 调用 `PytestRunner` 在子进程中运行生成的测试。
- 设置 `PYTHONPATH`，包含目标仓库根目录和 `src/`。
- 使用 `pytest-cov` 生成 coverage JSON。
- 记录每轮运行的 stdout、stderr、返回码和耗时。

pytest 命令大致为：

```text
python -m pytest generated_tests -q --cov=. --cov-report=term-missing --cov-report=json:coverage.json
```

### 4.7 Failure Analyst Agent

职责：

- 读取 pytest 输出。
- 将失败归因为：
  - `ImportError`
  - `FixtureError`
  - `AssertionError`
  - `EnvError`
  - `LogicMismatch`
- 生成 `FailureReport`，供 Repair Agent 使用。

### 4.8 Repair Agent

职责：

- 读取失败归因和生成测试文件。
- 调用 GLM 生成修复后的完整测试文件内容。
- 只允许替换 `test_*.py` 生成测试文件。
- 不修改业务源码。
- 输出必须是 JSON：

```json
{
  "files": {
    "test_unit_generated.py": "完整 Python 文件内容"
  }
}
```

## 五、编排模型

本项目是 Pipeline + 并行分叉 + 有限循环的混合编排。

### 5.1 Pipeline

串行阶段：

```text
Repo Scanner -> Spec Inference -> Test Planner
```

原因：

- 必须先扫描仓库，才能拿到目标符号。
- 必须先推断规格，才能规划测试。
- 必须先规划测试，才能进入 Writer 阶段。

### 5.2 并行分叉

并行阶段：

```text
Unit Writer + API Writer + Regression Writer
```

原因：

- 三类测试关注点不同。
- 三者写入不同输出文件。
- 并行可以降低整体等待时间。

### 5.3 有限修复循环

循环阶段：

```text
pytest -> failure analysis -> repair -> pytest
```

终止条件：

- pytest 通过。
- 达到 `--max-repair-rounds`，默认 2。
- 失败被标记为环境问题或不可恢复问题。

## 六、真实 LLM API 接入

`app/llm/zai_client.py` 封装真实 ZAI SDK：

```python
from zai import ZhipuAiClient

client = ZhipuAiClient(api_key=api_key)
response = client.chat.completions.create(
    model=model,
    messages=messages,
    temperature=temperature,
)
```

由于 SDK 调用是同步接口，项目使用 `asyncio.to_thread` 包装，避免阻塞事件循环。

并发控制：

```python
self._semaphore = asyncio.Semaphore(max_concurrency)

async with self._semaphore:
    response = await asyncio.to_thread(...)
```

默认参数：

- `--llm-model glm-4.7`
- `--llm-concurrency 5`
- `--llm-timeout 120`
- `--llm-retries 3`

也可以切换：

```bash
python -m app.main --repo . --llm-model glm-5 --llm-concurrency 5
```

## 八、报告内容

`ReportBuilder` 会输出：

- 仓库路径。
- 扫描到的 Python 文件数量。
- 已有测试文件数量。
- 目标符号数量。
- API 路由数量。
- GLM 推断出的行为规格。
- GLM 生成的测试计划。
- 生成测试文件列表。
- pytest 运行状态、返回码和耗时。
- coverage 覆盖率摘要。
- 失败归因。

- 每轮 pytest 执行历史。
- 最新 stdout/stderr。

## 十、设计边界

- Agent 系统只生成和修复测试，不修改业务源码。
- Repo Scanner 是确定性静态分析，不调用 LLM。
- Spec Inference、Test Planner、Writer、Repair 都由真实 GLM 驱动。
- 无 `ZAI_API_KEY` 或无 `zai` SDK 时直接失败。
- 模型输出不合法时，该 Agent 记录错误并进入报告。
- 并发统一由 `ZaiChatClient` 控制，默认最多 5 个 GLM 请求。

---

## 十一、项目亮点

1. **完整闭环**：不是只生成测试，而是覆盖扫描、规格、计划、生成、执行、归因、修复、报告。
3. **并行 Writer**：Unit/API/Regression 三类测试并行生成。
4. **并发限流**：共享 GLM Client 保证并发不超过模型额度。
5. **结构化状态**：使用 dataclass 管理跨 Agent 数据流。
6. **测试责任边界清晰**：Repair Agent 只修测试，不改业务源码。
7. **工程可观测**：报告包含计划、规格、coverage、失败归因和 Agent 事件。
